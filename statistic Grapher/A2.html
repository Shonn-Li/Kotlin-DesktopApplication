A2 Grapher

MVC, Drawing

Due Fri Feb 18 at 6 PM [60 marks]

Synopsis

You will implement a simple graphing application. It has multiple DataSets, each with a text title, text x- and y-axis labels, and a list of integer numbers between 0 and 100. When a DataSet is selected, a simple bar graph is drawn, basic statistics are shown, and the DataSet can be changed.

Watch a video demonstration of the a2 example solution (no audio).

Requirements and Grading

Make sure you have read and understand the course assignment policies.

You will submit two versions of your program:

  • In a2-basic, you implement as much required functionality as you can without any enhancements. Your goal is to match the video demonstration. This is what the markers evaluate for required functionality marks. Do this first.

  • In a2-enhanced, you can extend and deviate somewhat from the required user interface functionality to show off your programming skill, initiative to learn more related material, and design creativity. Do this second.

Technical Requirements (8 marks)

MVC Technical Requirements

  1. [2 marks] Your solution must include multiple view classes and a single model class. At minimum, there should be 6 main views representing the main parts of the application interface.

    HINT: You really only need these 6 views.

  2. [2 marks] The model should only communicate with views through the IView interface, and the model should not make any direct changes to the interface using any direct references to user interface widgets, panes, etc.

    HINT: A good rule of thumb is that the model class should have no JavaFX code and need no JavaFX-related imports.

  3. [2 marks] All synchronization of application data and state should be done through the model. A view does not communicate directly with any other view, and a view does not directly change another view (i.e. a view does not have a reference to any other view).

    CLARIFICATION: In the above, “view” refers to a class with an IView interface that registers with the model for notification. You may find it convenient to create “sub-view” functions or classes that a main view creates and manages, such as creating a small part of the main view.

  4. [2 marks] When a view modifies the model’s data or state, the model automatically handles the notification of all views (i.e. access to notifyObservers in the model must be private).

Additional Special Requirements for This Assignment

  • You must use the provided DataSet data class by including this file in your project: DataSet.kt

  • You may not use JavaFX graph widgets (i.e. you may not import anything in javafx.scene.chart)

Standard Requirements

  • Your implementation must be written in Kotlin 1.5 or higher, use the JavaFX toolkit, and run as a desktop OS application (Windows, MacOS, or Linux).

  • You may NOT use other toolkits (i.e. AWT and Swing are not allowed).

  • You may NOT use FXML, the interface must be built in Kotlin code.

  • You may use any code from the cs349 w22 public repo, as long as you acknowledge this clearly in your source code (a code comment is sufficient). As per course policy, you may not copy or use source code from any other source without explicit instructor permission.

  • You must include a README.md with at minimum: (1) your name, WatIAM username, and student number; (2) your development OS with version; (3) if you used IntelliJ, if so what version. Other development settings we can get from your project files, but if you want to also include the version of kotlin, Java SDK, and JavaFX you used, it could be helpful. Below all of this, add any additional information that the TA might need to grade it (e.g. an explanation of your enhancement for the a2-enhanced). For example:

  • Your code should use Gradle to manage builds: typing gradle build (or ./gradlew build) should build your program, and gradle run (or ./gradlew run) should run it. This should work when executed in the root of your a2-basic directory (and in your a2-enhanced directory if you chose to do that part).

  • Your project should be in a folder named a2 and the directory structure should look something like below (it does not need to match exactly, and will vary a little based on how it was created, and whether you choose to do a2-enhanced). Functional Requirements (50 marks)

This will be marked using a2-basic.

Each item is worth 2 marks, and they’re roughly ordered by difficulty. The final items are more challenging. Study the video to see these requirements demonstrated.

NOTE: interface dimensions are provided in JavaFX “units” as a guide, they may not necessarily be the same as pixels depending on the density and scaling of your OS and display. We do not measure pixels or JavaFX units when marking! For things like sizes, padding, and margins just follow the requirements and study the video. If a dimension isn’t given, study the video as a guide. Your goal is to achieve a layout very similar to the video.

  1. The application opens as a window named “A2 Grapher (userid)” where “userid” is your UWaterloo WatIAM username. The initial size of the application part of the window is 800 by 600 units when opened, but it can be resized with a minimum size of 600 by 400 units.

  2. There is a toolbar at the top and a LIGHTGREY status bar at the bottom. When the window height is resized, the ToolBar height and status bar height do not change.

  3. Below the toolbar is a fixed height horizontal interface area for editing the dataset title and axes, and below that are 3 interface areas: a 150 unit wide area for the data table; a centre WHITE area for the graph; and a 125 unit wide right side area for data statistics. When the window width is resized, only the width of the center white area changes in the middle band.

  4. The toolbar has a ChoiceBox (labelled “Dataset: ”), a New button, and a Spinner. The range of the Spinner is 1 to 20. The toolbar widgets are left-aligned and they do not wrap. There are 10 units of spacing around and between all toolbar widgets. The new button has a preferred width of 80 units (no wider). There is a vertical Divider between the ChoiceBox and the New Button.

    HINT: You can use the JavaFX ToolBar container widget for the toolbar.

  5. When the application first runs, the ChoiceBox in the toolbar lists the names of six default DataSets available for viewing: “Increasing”, “Large”, “Middle”, “Single”, “Range”, “Percentage”. Initially, the selected name in the ChoiceBox is “Increasing”

    HINT: The six default DataSets are generated with createTestDataSet in the provided DataSet.kt code.

    HINT: A Kotlin MutableMap is a good data structure to store DataSets in the model.

  6. The status bar displays the number of data sets available for viewing. When the application first loads it says “6 datasets”, and the number updates as new DataSets are added.

  7. The data statistics area (rightmost area in the application middle band) displays five statistics about the data in the currently selected DataSet: the “Number” of data points, the “Minimum” value, the “Maximum” value, the “Average” value, and the “Sum” of all data points.

  8. The data statistics area is formatted as 5 rows and 2 columns. Both columns are left-aligned. The “Average” is accurate to 1 decimal place.

  9. Below the toolbar is a dataset title and axes editing area. It has three TextFields labelled “Title:”, “X-Axis:”, and “Y-Axis:”. The TextFields contain the selected DataSet title, xAxis, and yAxis, and they may be edited.

  10. The data table area (left area in the application middle band) is a vertical list of each data point value in the currently selected DataSet.

  11. Each row of the data table area is a Spinner with a label showing the data point number (starting from 1). The list of values scrolls vertically if necessary, but it does not scroll horizontally. The padding, spacing, and width of the label and Spinner are selected so they are always visible with and without the vertical scrollbar.

  12. Selecting a different DataSet using the ChoiceBox in the toolbar changes the list of data point values in the data table area, changes the statistics in the statistics area, and changes the values in the edit dataset names and axis.

  13. In the data table area, the “up/down” buttons of each Spinner adjust the corresponding data value, and any resulting changes to the statistics are immediately visible. The Spinner range is 0 to 100 and the amount to step by is 1.

    HINT: An initial test is to select the “Single” dataset, then adjusting the single data point will change all statistics except “Number”.

  14. Pressing the “New” button in the toolbar generates a new random dataset named “NewX” (where X is a unique integer). A new random dataset has the number of data points indicated in the toolbar Spinner, with each data point randomly selected from the range 0 to 100. It has a random 3-word title (in Title Case) and random 1-word axes labels. The random words are taken from standard Lorem Ipsum text. The new dataset name is added to the toolbar ChoiceBox and selected.

    CLARIFICATION: You may use the LoremIpsum random text generator class in the A1 solution.

    Drawing

    IMPORTANT: You may not use the JavaFX graph widgets (i.e. you may not import anything injavafx.scene.chart).

  15. The central graph area displays an x- and y-axis as black lines. Behind the axes is a rectangle with LIGHTGRAY stroke (only the top and right lines are visible). There is a 50 unit margin around the axes and rectangle.

  16. There is a “0” labelling the bottom point of the y-axis and another number labelling the top of the y-axis.

  17. The text label for the x-axis is horizontally centred below the x-axis line and vertically centred in the bottom margin. A title for the graph is horizontally centered in the main graph area and vertically centred in the top margin.

  18. The label for the y-axis is vertically centred along the y-axis line and horizontally centred in the left margin. The y-axis label is rotated 90 degrees to read from bottom to top.

  19. The data points values are rendered as vertical bars. Each bar begins at the x-axis. Bars representing the maximum data point in the dataset extend to the top grey line of the rectangle (i.e. the range of the y-axis is 0 to the maximum data point value)

  20. The bars have equal width and are spaced apart by 5 units. There is a 5 unit space between the leftmost bar and the y-axis and a 5 unit space from the right-most bar and the right grey border line. The bar fill colors form a rainbow.

    HINT: A rainbow is a uniform distribution of Hue in the HSB colour model. You can use Color.hsb with saturation and brightness 1.0.

  21. The top number labelling the top of the y-axis is the maximum data point in the dataset.

  22. Editing the TextFields in the dataset title and axes area changes the corresponding text in the graph. The edits take effect immediately (i.e. after each keystroke).

  23. Editing any datapoint in the data table form immediately changes the data in the graph, including the maximum point label in the y-axis and the upper limit of the y-axis range.

  24. Any edits to the selected dataset title, axis labels, and any edits to data point values remain in that dataset while the application is running (i.e. if another dataset is selected, then the changed dataset is selected again, the edited values are still there).

  25. The graph resizes to fill the centre graph area of the interface without any cropping. The margin size, line thickness, font size, and spacing between bars do not change. The length of the axes, length of the bars, and thickness of the bars change to fill the space.

Enhancement (2 marks)

This will be marked using a2-enhanced.

Once you have required functionality above working, enhance the functionality or the visual design in a version of the user interface in a program called a2-enhanced.

These basic requirements must still be followed:

  • The application should still be a graphing application with the ability to display and change different data sets.
  • The application should still have the same high level layout (a toolbar at top, graph manipulation and information interface in the middle, and status bar at bottom). You can change what’s in these parts of the interface and you can introduce other interface areas.
  • The window must still be able to be resized with reasonable responsive adjustment in the interface, and it should look fine at the initial resolution specified above.
  • The enhancement should be visible through the user interface and be directly related to the user interface. For example, adding a database backend, or scraping data from a website would not count as enhancements unless there was a significant user interface aspect.

Some enhancement ideas:

  • Add a way to view a DataSet using another type of graph such as a pie chart.
  • Add the ability to graph 2 datasets on the bar chart (using different colours with a legend)
  • Add an interface to edit the layout of the bar chart: change the margin, bar colours, font size, etc.
  • Make the graph interactive, for example hover over a bar to see the value, click on bars to fade them out to allow comparing specific data points, etc.

Any one of the bullets above would be a fine enhancement, but you’re encouraged to dream up your own enhancements too. Those bullets are just examples so you get a sense of “how big” an enhancement should be.

You must describe your enhanced version in your README.md file (see above). The markers need to know how to use your program so they can see how great it is.

It’s OK to not do the enhanced version, or to keep it minimal if you’re running out of time. Remember, it’s only worth a small portion of the assignment.

General Deductions

Up to 5 marks could be deducted if:

  • Source code is very sparsely commented.
  • Formatting is very inconsistent (indent, spacing).
  • Classes, methods, and fields have cryptic, very short, or unclear names.
  • The code has obviously duplicated sections that could have been refactored.
  • The code is generally poorly unorganized. For example, multiple large class definitions are in one file.
  • Many unnecessary files are submitted, like system and build-generated files (e.g. .DS_Store).

You could lose more than 5 marks if your code doesn’t compile or crashes easily.

Submission

Submit your assignment to your personal CS 349 repository in the a2/ subdirectory.

Ensure all files required to compile the program are checked in. It’s your responsibility to check your submission (e.g. by cloning it to a fresh directory and testing it like a marker will do).
